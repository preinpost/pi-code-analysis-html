InfluxDB, 백엔드에서 어떻게 쓰이고 있나

OpenStack 노드·인스턴스·GPU의 시계열 메트릭 저장소. 실제 운영은 InfluxDB v1.8이고, 백엔드는 v2 클라이언트의 v1 호환 모드로 접속해 Flux 쿼리로 읽는다. 쓰기는 LB 메트릭 한 곳뿐 — 전형적인 “에이전트가 수집, 백엔드가 조회” 구조다.

01

데이터 흐름 — 수집은 에이전트, 백엔드는 조회

애플리케이션은 메트릭을 직접 쓰지 않는다. compute 노드의 telegraf/hsflowd가 InfluxDB에 적재하고, 백엔드는 Flux DSL로 쿼리를 조립해 getQueryApi()로 읽는다.

유일한 쓰기 경로 — LB 메트릭. LoadBalancerMetricServiceImpl이 WriteApi.writeMeasurements(MS) 로 load_balancer_metric 을 역기록한다. batch 쪽 List<WriteApi> 빈을 통해 모든 노드에 중복 기록. 나머지 메트릭은 앱이 쓰지 않는다.

02

모듈 의존성 — InfluxDB 코드가 놓이는 자리

03

관련 프로그램 — 어디가 InfluxDB를 만지나

모듈별 관여도 한눈에 보기. common과 batch가 실제 쿼리·쓰기를 담당하고, core는 설정·DTO, servlet은 읽기 전용 구성만 갖는다.

04

파일 의존성 그래프 — 모든 파일의 연결 지도

InfluxDB를 만지는 모든 파일을 노드로 펼쳤다. 실선은 컴파일 의존, 점선은 런타임 빈 주입, 크라프트선은 InfluxDB로의 조회/쓰기. 노드에 마우스를 올리면 해당 파일의 연결만 강조된다.

uses (컴파일 의존) bean 주입 (런타임) InfluxDB 조회/쓰기 설치/배포 노드 호버 → 연결 강조
05

클라이언트 구성 — servlet vs batch

같은 InfluxDB2Properties 를 읽지만 두 모듈의 빈 구성이 다르다. batch 가 다중 노드·쓰기 지원으로 더 진화한 형태.

08

관찰 & 개선 포인트

동작에는 문제없지만 정리할 잔재와 불일치가 있다.

01
v1/v2 혼재 잔재

실운영은 v1.8 + v2 클라이언트 호환 모드. 주석 처리된 v2 코드와 레거시 influxdb-java:2.25 의존성이 남아 있다.

02
servlet / batch 구성 불일치

servlet 은 단일 createV1, batch 는 다중 클라이언트 랜덤. 한쪽으로 통일할 필요가 있다.

03
InfluxDB2Util 사문화

수동 리플렉션 변환기가 query(String, Class) 자동 매핑으로 대체됨. 정리 후보.

04
influxDBClients.get(0) 하드코딩

batch metering 이 랜덤 빈 대신 0번 클라이언트 고정 사용 — 다중 노드 구성의 이점을 못 살린다.

05
에러 처리 취약

convertFluxTableToPOJO 가 예외를 silently ignore. 실패 시 빈 결과로 조용히 넘어간다.
